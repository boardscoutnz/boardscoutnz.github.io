Standing instructions for Claude in this repo. Read this first; then jump to the
relevant `docs/` topic file rather than re-reading source.

## 1. Project orientation

Board Scout NZ (BSNZ) is a single-user static site published at
`boardscoutnz.github.io` that joins a Trade Me board-game listings snapshot with
a BoardGameGeek rankings cache in a sortable, filterable Tabulator grid. Data is
collected by two Tampermonkey userscripts (`tprmky/`) and shipped as static JSON
under `data/`; the site itself does no uploads.

Canonical docs index ‚Üí [`docs/Board_Scout_NZ___Project_Overview.md`](docs/Board_Scout_NZ___Project_Overview.md).
The 12 topic files (`docs/01-stack.md` ‚Ä¶ `docs/12-debug-system.md`) are the
source of truth for architecture and gotchas ‚Äî do not duplicate their content
here.

## 4. Build commands

Two builds, both pure concatenation. Neither has dependencies; both are idempotent.

| Script | Output | Run when |
|---|---|---|
| `bash components/build.sh` | `index.html` | After any change to `components/*.html`, **or** when adding/removing files in `css/` or `js/` |
| `bash tprmky/build.sh` | `tprmky/tm-bgbf.user.js` | After any change to `tprmky/tm-bgbf-src/*.js` |

**JS-only or CSS-only edits to existing files do NOT require a rebuild** ‚Äî they
are referenced directly by the already-generated `index.html`. Just reload the
browser. Rebuild is only needed when the *list* of `css/` or `js/` files
changes, or when an HTML fragment changes.

The `bgg-ranks-exporter.user.js` script has no build ‚Äî edit it directly.

## 5. Coding conventions

Project-specific rules only ‚Äî generic style is fine.

- **`js/` is classic-script shared scope.** All `js/*.js` files are loaded as
  `<script defer>` (not `type="module"`) and share the document's lexical
  scope. Do **not** wrap modules in IIFEs, do **not** convert to ES modules,
  and do **not** use top-level `var`. Top-level `let`/`const`/`function` is the
  pattern; cross-module references resolve at call time.
- **CSS cascade is filename-ordered.** Numeric prefixes drive load order via
  `components/build.sh`'s glob. `04-sidebar.css` MUST load before
  `12-responsive.css` so the responsive sidebar media query overrides the base
  rules. Don't renumber files casually.
- **Userscript IIFE is split across files.** `tprmky/tm-bgbf-src/00-header.js`
  opens `(function () { 'use strict'; ‚Ä¶` and `99-footer.js` closes `})();`.
  Files `01-*.js` ‚Ä¶ `16-*.js` are *fragments* of statements inside that IIFE
  and are **not standalone valid JavaScript** ‚Äî `node --check` on a body file
  will report syntax errors and that's expected. Validate the concatenated
  output instead.
- **Do not add `'use strict';` to userscript body files.** It's already in
  `00-header.js`; a duplicate inside the IIFE is a syntax error.
- **`js/app.js` is the master + sanity check.** It is loaded last and verifies
  that 9 critical entry points (`APP_VERSION`, `listings`, `loadBggCache`,
  `matchTitle`, `loadListings`, `showGrid`, `applyFilters`,
  `wireToolbarControls`, `showToast`) are defined. If a child module is missing
  or out of order it logs `[bsnz] app.js master sanity check FAILED ‚Äî missing:
  [‚Ä¶]`. If you see that error after a change, you've broken the load order.
- **Tabulator gotchas are dense.** Before touching grid, sort, filter, or BGG
  mode code, read [`docs/08-gotchas.md`](docs/08-gotchas.md).

## 6. Commit conventions

- Short imperative subject lines (e.g. `fix sidebar overflow on narrow viewports`).
- **Always commit the rebuilt artefact alongside its source change**:
  - `components/`, `css/` (file added/removed), or `js/` (file added/removed)
    edits ‚Üí also commit `index.html`.
  - `tprmky/tm-bgbf-src/` edits ‚Üí also commit `tprmky/tm-bgbf.user.js`.
- Branch names: `fix/<short-slug>`, `feature/<short-slug>`, `chore/<short-slug>`.
- Don't mix unrelated subsystems in one commit.

## 7. Verification before declaring done

Run the matching checks for every change. All must exit 0.

- [ ] **JS edits** ‚Äî `node --check js/<file>.js` for every touched `js/*.js`.
- [ ] **CSS edits** ‚Äî visual smoke check; no automated check.
- [ ] **HTML component edits** ‚Äî `bash components/build.sh`, confirm exit 0,
      confirm `index.html` regenerated, commit it.
- [ ] **Userscript edits** ‚Äî `bash tprmky/build.sh`, confirm exit 0, confirm
      `tprmky/tm-bgbf.user.js` regenerated, commit it.
- [ ] **Sanity check** ‚Äî if the change touches script load order or adds/removes
      a `js/*.js` file, load the site locally and confirm no `[bsnz] app.js
      master sanity check FAILED` error in the console.

For UI-affecting changes, state explicitly that you have **not** browser-tested
unless you actually started a server and used the feature.

## 8. Out of scope (do not touch unless explicitly asked)

- `img/` ‚Äî image assets.
- `data/*.json` ‚Äî `listings.json`, `bgg-rankings.json`, samples. These are
  produced by the userscripts; never hand-edit.
- `site.webmanifest` ‚Äî PWA manifest.
- `.github/` ‚Äî repository/CI configuration.
- `index.html` ‚Äî build artefact (see ¬ß2).
- `tprmky/tm-bgbf.user.js` ‚Äî build artefact (see ¬ß2).

If a task seems to require touching one of these, stop and ask.

## 9. Default behaviour for ambiguous requests

1. Read the relevant `docs/NN-*.md` topic file and the 1‚Äì2 source files from
   the ¬ß3 lookup table **before** proposing changes.
2. If scope is genuinely ambiguous, ask **one** clarifying question. Don't
   front-load a list.
3. If the change will touch more than 2 files, propose a plan and wait for
   approval before editing.
4. Default to minimal, surgical changes. No drive-by refactors, no new
   abstractions, no reformatting.
